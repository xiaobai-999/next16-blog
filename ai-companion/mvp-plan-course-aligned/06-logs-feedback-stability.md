## 工程约定

- 每次聊天请求入口生成 `trace_id`。
- `trace_id` 贯穿聊天、模型调用、消息保存、记忆提取。
- MVP 使用 D1 `model_logs` + Workers `console.log`。
- 日志不保存完整敏感 prompt。
- `GET /debug/model-logs` 只允许本地或管理员访问。


第一版如果拿不到精确 token，可以先允许为空。

## API 边界

```text
POST /feedback
GET  /debug/model-logs
```

`GET /debug/model-logs` 第一版只建议本地或管理员可用，不开放给普通用户。

## 反馈请求

```json
{
  "messageId": "assistant-message-id",
  "rating": "up",
  "reason": "回答有帮助"
}
```

`rating` 可选值：

```text
up
down
```

## 反馈权限

- 只能反馈自己的消息。
- 只能反馈 `assistant` 消息。
- 同一个用户对同一条消息第一版可以允许覆盖，也可以拒绝重复提交；默认采用覆盖策略。

## 错误处理策略

聊天接口至少处理：

- 用户未登录
- 用户没有 companion
- conversation 不属于当前用户
- 模型 API 调用失败
- 模型返回超时
- 数据库保存失败
- 记忆提取失败

推荐规则：

```text
聊天主回复成功后，即使记忆提取失败，也不要影响用户收到回复。
模型调用失败时，保存失败日志，并给用户一个可重试的错误状态。
消息保存失败时记录 trace_id，前端提示当前回复可能未保存。
```

本次实现采用的 MVP 策略：

- 聊天入口生成 `trace_id`，响应头返回 `X-Trace-Id`，方便前端和 Workers 日志对齐。
- 主聊天模型调用和记忆提取模型调用都写入 `model_logs`，只记录模型元信息、token、耗时和错误摘要，不记录完整 prompt。
- 主聊天回复成功后，记忆提取失败只写日志，不影响用户收到 assistant 回复。
- `POST /feedback` 只允许登录用户反馈自己的 assistant 消息，同一用户对同一消息重复反馈采用覆盖策略。
- `GET /debug/model-logs` 需要登录，且第一版只允许从 `localhost`、`127.0.0.1` 或 `::1` 访问。
