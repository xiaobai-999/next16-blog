# 项目问题复盘：长期记忆没有保存

## 背景

项目中实现了 AI 伴侣的长期记忆能力。预期行为是：用户在聊天中明确表达可复用偏好，例如“请记住：我不喜欢吃香菜，以后推荐吃的不要包含香菜”，系统会在 assistant 回复完成后异步提取记忆，并写入长期记忆库，随后可以在 `/memories` 页面看到。

## 问题现象

用户发送了明确的记忆触发语句，但 `/memories` 页面没有出现已保存记忆，也没有待确认记忆。

一开始容易误判为：

- 用户表达不够明确，没有触发记忆提取。
- `/memories` 页面展示逻辑有问题。
- 记忆被保存成了其他状态，比如 `pending_confirmation`。

但实际排查后发现，聊天回复本身是成功的，失败发生在聊天结束后的“自动记忆提取”阶段。

## 排查路径

我按完整链路排查，而不是只看前端页面：

1. 确认用户消息是否正常保存。
2. 确认 assistant 回复是否正常保存。
3. 查看聊天完成后是否调用 `extractAndStoreMemories`。
4. 查看后端日志中记忆提取阶段是否报错。
5. 再确认 `/memories` 查询逻辑是否只展示 `active` 和 `pending_confirmation`。

最终在后端日志中看到错误：

```text
AI_APICallError: This response_format type is unavailable now
```

这个错误说明问题不在数据库，也不在 `/memories` 页面，而是在模型执行结构化记忆提取时失败了。

## 根因

原实现使用 AI SDK 的 `generateObject` 做记忆提取。

`generateObject` 会依赖模型服务商的结构化输出能力，底层通常会使用类似 `response_format` 的参数，让模型按 JSON schema 返回对象。

但当前项目聊天模型使用的是 DeepSeek 的 OpenAI-compatible 接口。OpenAI-compatible 只代表基础 API 形态相似，并不代表完全支持 OpenAI 的全部高级参数。当前接口返回：

```text
This response_format type is unavailable now
```

所以模型无法完成结构化输出，候选记忆没有生成，后续的记忆写入逻辑自然不会执行。

## 修复方案

将记忆提取从强依赖结构化输出，改成更兼容的文本 JSON 方案：

1. 使用 `generateText` 代替 `generateObject`。
2. 在 system prompt 中明确要求模型只输出 JSON 对象。
3. 要求输出格式为：

```json
{"memories":[...]}
```

4. 服务端收到文本后，先剥离可能出现的 Markdown code fence。
5. 再执行 `JSON.parse`。
6. 最后仍然用 Zod schema 校验结构，保证入库数据可靠。

这样做的好处是：

- 不再依赖 provider 是否支持 `response_format`。
- 保留了结构化校验能力。
- 聊天模型仍然可以继续使用 DeepSeek。
- 自动记忆提取对模型服务商更兼容。

## 关键代码思路

```ts
const result = await generateText({
  model: getChatModel(env),
  system: MEMORY_EXTRACTOR_SYSTEM_PROMPT,
  prompt,
  temperature: 0
});

const parsed = parseMemoryExtractionText(result.text);
candidateMemories = parsed.memories;
```

解析函数负责兼容模型偶尔输出代码块的情况：

```ts
function stripJsonFence(text: string) {
  const trimmed = text.trim();
  const fenced = trimmed.match(/^```(?:json)?\s*([\s\S]*?)\s*```$/i);

  return fenced?.[1]?.trim() ?? trimmed;
}
```

## 面试表达

这个问题我会这样向面试官解释：

在做 AI 伴侣长期记忆功能时，我遇到过用户明确说“请记住我不喜欢吃香菜”，但记忆管理页没有新增记录的问题。

我没有直接把它归因于 prompt 没写好，而是沿着链路排查：用户消息保存、assistant 消息保存、记忆提取、记忆写入、记忆列表查询。最后发现聊天生成是成功的，但记忆提取阶段失败，日志里有 `response_format type is unavailable now`。

根因是原来用 AI SDK 的 `generateObject` 做结构化记忆抽取，它依赖模型服务商支持 `response_format`。但项目当时接入的是 DeepSeek 的 OpenAI-compatible 接口，这个接口并不完全支持相关结构化输出能力，所以候选记忆没有生成。

我的修复方式是把这一步改成 provider-agnostic 的方案：用 `generateText` 让模型输出 JSON 文本，然后服务端本地做 JSON 解析和 Zod schema 校验。这样既避免了对特定 provider 高级参数的依赖，又保留了结构化数据的可靠性。

这个问题让我意识到，AI 应用中的“模型调用成功”并不等于所有能力都可用。聊天生成、结构化输出、embedding、工具调用是不同能力面。做工程实现时，需要对 provider 能力做分层判断，并为关键路径设计兼容方案或 fallback。

## 可总结的工程经验

- OpenAI-compatible 不等于完全兼容 OpenAI 的所有高级能力。
- 自动记忆提取属于结构化抽取任务，不应只按普通聊天任务看待。
- AI 应用排障要按链路拆解：生成、抽取、校验、写入、查询分别确认。
- 对长期记忆这类关键能力，失败不能影响主聊天，但必须记录日志，方便追踪。
- 结构化输出最好保留服务端 schema 校验，不能只相信模型返回。
